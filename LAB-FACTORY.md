# LAB-FACTORY — how to add lab #3..#15

This is the definitive guide to authoring a new ShieldSync AWS security lab. It was
written by reading the two live labs end-to-end (`s3-misconfiguration-audit`,
`iam-privilege-escalation`) — every convention documented here is copied from what
those two labs actually do, not invented. Where the platform has a rough edge (mostly:
duplication and hardcoded per-slug dispatch), that's called out explicitly rather than
smoothed over, so the next author isn't surprised by it mid-build.

Companion skeleton: `labs-platform/labs/_template/` — copy it as your starting point.

---

## 1. Anatomy of a lab — every artifact, exact path

A "lab" is currently **eight files across three directories**, plus one registry
dispatch entry in the grader and (optionally) three marketing-site touch points.
There is no single manifest that lists them — this table *is* that manifest.

| # | Artifact | Exact path | Purpose |
|---|---|---|---|
| 1 | CFN scenario template | `labs-platform/labs/<slug>/template.yaml` | Deploys the deliberately-broken mini-environment. Consumed by `engine/labinfra.mjs` (`deployStack`, `deployLab`, `warmAccount`). |
| 2 | Engine lab.json (policy source of truth) | `labs-platform/labs/<slug>/lab.json` | The copy the **engine** reads at runtime (bundled into the Lambda zip). Supplies `level`/`free` (access rules), and — critically — `learnerPolicy` (session-policy scoping). Read by `labinfra.mjs` (`labMeta`, `labLearnerPolicy`) at lease/console-mint time. |
| 3 | App lab.json (content source of truth) | `labs-platform/app/content/labs/<slug>/lab.json` | A **second, separate copy**, hand-authored alongside #2. Supplies `title`/`summary`/`tags`/`estimatedActiveMinutes`/`successCriteria`/`hints` — the marketing/UI metadata and the *objectives* shown to the learner. Consumed by `app/scripts/build-lab-content.mjs`, NOT by the engine. |
| 4 | Instructions markdown | `labs-platform/app/content/labs/<slug>/instructions.md` | The learner-facing walkthrough. Must follow the sentinel/step/track conventions in §4. Consumed by the same build script. |
| 5 | Generated catalog (do not hand-edit) | `labs-platform/app/lib/lab-catalog.ts` | Regenerated by `node scripts/build-lab-content.mjs` (run from `app/`) from artifact #3 + whether artifact #1 exists (`ready` flag). |
| 6 | Generated content (do not hand-edit) | `labs-platform/app/lib/lab-content.ts` | Regenerated by the same script from artifact #4 (instructions) + artifact #3's `successCriteria` (objectives). |
| 7 | Grader dispatch entry | `labs-platform/engine/graders.mjs` | **Hardcoded if/else** in `gradeLab()` — see §2. There is no generic "check successCriteria against account state" engine; you write a `grade<Lab>()` function by hand and wire it in. |
| 8 | Marketing catalog entry | `shieldsync-website/lib/site.ts` → `AWS_LABS` array | A **third, independent, hand-authored copy** of slug/title/level/free/tags/minutes (plus marketing-only `desc`/`added`/`skills`/`roles`). Drives the labs listing, sitemap, and SEO/OG metadata on the marketing site. See §5 for the full H11 catalog-duplication map. |

### Files you author by hand (5 of 8)
`template.yaml`, both `lab.json` copies, `instructions.md`, `graders.mjs`'s new
function + dispatch line, and the `AWS_LABS` entry. The other three
(`lab-catalog.ts`, `lab-content.ts`) are generated — never hand-edit them, your
changes will be silently overwritten on the next `build-lab-content.mjs` run.

### Naming / tagging conventions the engine expects (read from the S3 + IAM templates)

- **`LabSlug` parameter.** Every template declares `Parameters: LabSlug: {Type: String, Default: <slug>}` — the default MUST equal the directory name (it's how `deployLab`'s stack-name and the tag value line up; nothing enforces this at deploy time, a mismatch just silently mistags resources).
- **`ShieldSyncLab` tag.** Every taggable resource carries `Tags: [{Key: ShieldSyncLab, Value: !Ref LabSlug}]`. This is what `aws-nuke`'s per-account teardown config and the account-level `control-plane` preset use to leave engine-owned roles alone while nuking everything the lab created. **Any resource type that does NOT support CloudFormation tags** (IAM's `ManagedPolicy` and `AccessKey`, confirmed in `iam-privilege-escalation/template.yaml`) is instead targeted by aws-nuke via name/path prefix convention (see next point) — the account-level nuke is the backstop if that also misses.
- **Resource naming prefix.** S3 buckets: `sslab-<purpose>-${AWS::AccountId}` (e.g. `sslab-data-${AWS::AccountId}`). IAM users/roles the learner interacts with: `Path: /lab/` (e.g. `/lab/auditor`, `/lab/pipeline-deployer`) — this path is ALSO what the `learnerPolicy`'s `IamLabUser` statement scopes to (`arn:aws:iam::*:user/lab/*`), so the path isn't cosmetic, it's the IAM scoping boundary. Seeder Lambda / role names: `sslab-<slug>-seeder-${AWS::Region}`.
- **`DeletionPolicy: Delete` + `UpdateReplacePolicy: Delete`** on every resource. `deployStack` creates with `OnFailure: DELETE` so a failed deploy leaves no debris and the account recycles cleanly.
- **Region-agnostic, `us-east-1` default.** Both labs declare `deploysToRegion: "us-east-1"` in lab.json and the engine's `REGION` constant is fixed to `us-east-1` — don't author a template that assumes a different region.
- **No orphaned objects on teardown.** If your scenario writes objects into an S3 bucket (seed data), you need a custom-resource Lambda that **empties the bucket on Delete** (see `SeederFunction`'s `empty()` in the S3 template) — otherwise `DeletionPolicy: Delete` cannot actually delete a non-empty bucket, and the stack teardown wedges (the account-level aws-nuke is the backstop, but that's a much slower/uglier failure mode than a clean stack delete).
- **Cost.** Both existing labs declare `estimatedAwsCostInr: 1` — keep new labs near-zero cost (no idle compute; a seeding Lambda invoked once via custom resource is the established pattern for "the account needs some pre-existing state").
- **Outputs.** Both templates end with an `Outputs:` block surfacing the real resource names/values the instructions.md placeholders reference (e.g. `DataBucketName`, `AttackerAccessKeyId`) plus a convenience `*ConsoleUrl` output. `deployStack` collects these into `outputs` — currently nothing in the app auto-injects them into the instructions markdown (the instructions.md just tells the learner "the Session Engine filled in your real names" and shows a placeholder table); confirm whether your lab needs the same manual-table pattern or should account for this gap.

### Registration points — every place a new slug must be added

This is the answer to "I wrote all 8 files, why doesn't my lab show up / work end to end":

1. **`labs-platform/engine/graders.mjs`** — add a `grade<YourLab>()` function and a new `else if (labSlug === "<slug>") criteria = await grade<YourLab>(...)` branch inside `gradeLab()`. Skip this and "Check my work" silently returns `{gradable: false, passed: false}` for your lab — the grade button will always show as if nothing passed.
2. **`labs-platform/app/lib/lab-catalog.ts` and `lab-content.ts`** — NOT edited directly; regenerated by running `node scripts/build-lab-content.mjs` from `labs-platform/app/`. Skip this and your lab never appears in `LABS` (the catalog `app/lib/labs.ts` re-exports from `lab-catalog.ts`), so `/labs/<slug>` 404s regardless of what's in `content/labs/<slug>/`.
3. **`shieldsync-website/lib/site.ts` → `AWS_LABS`** — hand-add an entry (mirroring slug/title/level/free/tags/minutes from the app's lab.json, per the comment already in that file: *"CANONICAL shared fields ... mirror each lab's labs-platform/app/content/labs/<slug>/lab.json — keep them in sync"*). Skip this and the lab exists and works on `labs.shieldsyncsecurity.com` but is invisible on the marketing site (no listing card, no sitemap entry, no SEO landing, no OG/schema.org `Course` markup pointing at it — the schema.org JSON-LD IS generated per-lab automatically by `app/labs/[slug]/page.tsx`, but only for labs already in the catalog it iterates, which traces back to `AWS_LABS` for the marketing pages and to `lab-catalog.ts` for the app's own `/labs/[slug]` route).
4. **(Cosmetic, easy to miss, low stakes) `shieldsync-website/components/labs-wizard.tsx`** — `const FREE_SLUG = "s3-misconfiguration-audit"` is hardcoded. Only matters if your new lab is meant to be the free one (unlikely for lab #3+, since there's already a free lab) — otherwise no action needed.
5. **(Cosmetic) `shieldsync-website/app/aws-security-certification/page.tsx` line ~264** — a hardcoded `l.slug === "s3-misconfiguration-audit"` check that renders a "FREE" pill. Same caveat as #4 — only relevant if your new lab is free.
6. **(Optional, UX-only) `labs-platform/engine/handler.mjs` / `server.mjs` → `PRIMARY_LAB`** — hardcoded to `"s3-misconfiguration-audit"`, used only by the background warmer (`ensureWarm`) that pre-deploys the free lab into idle pool accounts so its *first* launch is instant. Not adding your slug here does not break the lab — it just means every launch of your new lab is a cold deploy (~80-110s) instead of instant. Only relevant if you want your new lab warmed too (would need `ensureWarm` extended to warm multiple labs, which it doesn't do today — currently it's a single hardcoded slug, not a list).

Everything else (`deployLab`, `mintConsoleUrl`, `teardown`, `lease`, `rulesFor`,
`labLearnerPolicy`) is already slug-generic — it takes `labSlug` as a runtime
parameter and reads `labs/<slug>/{lab.json,template.yaml}` off disk. You do not need
to touch `labinfra.mjs` to add a lab.

## 2. The grader model — how objectives map to live-account checks

**This is the single most important thing to understand before authoring lab #3.**
The grader is **not data-driven**. `successCriteria` in lab.json is *documentation and
UI copy* (it's what renders as the objectives checklist in `LabPanel`) — it is **not**
executed. The actual pass/fail logic is hand-written imperative code per lab in
`engine/graders.mjs`:

```js
export async function gradeLab(labSlug, execRoleArn, accountId) {
  const c = await assumeInSandbox(execRoleArn, "engine-grade");
  const creds = { accessKeyId: c.AccessKeyId, secretAccessKey: c.SecretAccessKey, sessionToken: c.SessionToken };
  let criteria = null;
  if (labSlug === "s3-misconfiguration-audit") criteria = await gradeS3(creds, accountId);
  else if (labSlug === "iam-privilege-escalation") criteria = await gradeIam(creds);
  if (!criteria) return { gradable: false, criteria: [], passed: false };
  return { gradable: true, criteria, passed: criteria.every((x) => x.passed === true && !x.unknown) };
}
```

For lab #3 you add a third `else if` and a new `gradeYourLab(creds, accountId)`
function. **The `id`s you return from that function must exactly match the `id`s in
that lab's `successCriteria`** in the app-side lab.json — that's the only linkage
between the two; nothing type-checks it, a typo just makes an objective permanently
show as un-graded in the UI (it won't error, `objectives` and grader `criteria` are
zipped together client-side by `id`).

### The pattern every `grade<Lab>()` function follows

1. **Assume the exec role** (done once by the caller, `gradeLab`) — read-only creds into the learner's sandbox account.
2. **Probe live AWS state** with the SDK — `S3Client`/`IAMClient` calls like `GetBucketPolicyStatusCommand`, `ListAttachedUserPoliciesCommand`, `GetPolicyVersionCommand`. Every probe is wrapped in try/catch.
3. **Distinguish "fixed" from "broken" from "couldn't check".** This is the subtle part — study `ABSENCE_ERRORS`/`isAbsenceError()`/`unk()` in graders.mjs:
   - An error like `NoSuchBucketPolicy` or `NoSuchEntity` means "the thing legitimately doesn't exist" — which for most of these labs **is the fixed state** (e.g. no bucket policy → not public). This is NOT a grading failure.
   - Any OTHER error (throttling, transient `AccessDenied`, network) means the **check itself failed** — the criterion is marked `{ ...unk(err) }` → `{ unknown: true }`. An `unknown` criterion is never counted as a pass (`gradeLab`'s final `passed` requires `x.passed === true && !x.unknown`), specifically so a transient AWS blip never reads to the learner as "you fixed it" when the check simply didn't run.
4. **Return `[{ id, description, passed, ...unk(err) }]`** — one object per objective, matching `successCriteria` ids.
5. **Helper predicates worth reusing:** `parsePolicy(doc)` (handles both IAM's URL-encoded policy documents and S3's plain JSON), `allowsActionOnStar(policy, testFn)` (does any Allow statement grant a matching action on `Resource: "*"`), `hasDeny(policy, predicateFn)` (does any Deny statement match a predicate — used for the encryption-required / TLS-only checks by testing the statement's `Condition` block).

### Grading a "prove you did the attack, then fix it" lab (like IAM priv-esc)

`gradeIam` shows the pattern for a two-phase lab: check the exploit was
**closed** (`admin-detached`, `escalation-primitive-removed`) AND that remediation
didn't **break legitimate function** (`deployer-still-works` — checks the user's
original read permission is still granted, by reading policy documents directly
rather than `iam:SimulatePrincipalPolicy`, because the simulator is eventually
consistent and returned false negatives against a freshly-attached policy in testing).
If your lab has an "attacker seat" + "defender seat" split (leaked keys), model your
`learnerPolicy` accordingly (§3) — the grader only ever assumes the **exec role**
(full account visibility), it does not need to know about the attacker keys at all,
since it inspects account state directly rather than replaying the learner's actions.

### What "the grader model" does NOT give you

- No generic "GetBucketEncryption == true → pass" declarative check runner.
- No way to grade non-S3/IAM services without writing new SDK client imports into `graders.mjs` (it currently only imports `@aws-sdk/client-s3` and `@aws-sdk/client-iam`) — a lab touching EC2, Lambda, KMS, etc. needs its SDK client added at the top of the file.
- No sandboxing between labs' grader code — all `grade<Lab>()` functions live in one file. At 15 labs this file will be several hundred lines of hand-written probes; see §5 for the consolidation note.

## 3. The learnerPolicy model — least-privilege session scoping

Every lab's console session is fenced with an **STS session policy**, not a per-lab
IAM role. The mechanics (all in `engine/labinfra.mjs`):

1. There is ONE role in every sandbox account the learner ever assumes: `ShieldSyncLabUser` (broad/admin-ish by design — it's the ceiling).
2. At console-mint time (`mintConsoleUrl`), the engine reads `labLearnerPolicy(labSlug, accountId)`, which loads `learnerPolicy` from the **engine-side** `labs/<slug>/lab.json` (artifact #2, NOT the app-side copy) and appends a **fixed, non-negotiable `guardrailDeny`**:
   ```js
   function guardrailDeny(accountId) {
     return [
       { Sid: "ssGuardrailProtectControlPlane", Effect: "Deny", Action: ["iam:*", "sts:AssumeRole"],
         Resource: [`arn:aws:iam::${accountId}:role/ShieldSyncLab*`, `arn:aws:iam::${accountId}:role/OrganizationAccountAccessRole`] },
       { Sid: "ssGuardrailDenyGovernance", Effect: "Deny", Action: ["organizations:*", "account:*"], Resource: "*" },
     ];
   }
   ```
   This guardrail is appended to **every** lab automatically — you never author it, and it means a learner can never touch the engine's own control-plane roles or org governance even if your `learnerPolicy` were authored too broadly.
3. The resulting document is passed as `Policy` on `AssumeRoleCommand` (STS session policy). **Effective permissions = `ShieldSyncLabUser` role policy ∩ your `learnerPolicy` ∩ account SCPs.** The role stays broad; the *session* is what's scoped down to exactly this lab.
4. If a lab has NO `learnerPolicy` (or an empty one), `mintConsoleUrl` mints an **UNSCOPED** admin-within-SCP console and only logs a warning — it does NOT block the mint at runtime. The build-time gate is what actually enforces this (see next point) — don't rely on a bad `lab.json` failing loudly at launch time.

### The authoring shape

```json
"_learnerPolicy": "LEAST-PRIVILEGE: bare IAM Statement[] applied as an STS SESSION POLICY on the learner's console ...",
"learnerPolicy": [
  { "Sid": "CloudShell", "Effect": "Allow", "Action": "cloudshell:*", "Resource": "*" },
  { "Sid": "Identity", "Effect": "Allow", "Action": "sts:GetCallerIdentity", "Resource": "*" },
  { "Sid": "...", "Effect": "Allow", "Action": "...", "Resource": "..." }
]
```
- `learnerPolicy` is a **bare `Statement[]` array**, not a full `{Version, Statement}` document — the engine wraps it in the `Version` envelope for you (`{ Version: "2012-10-17", Statement: [...statements, ...guardrailDeny(accountId)] }`).
- **Both existing labs grant `CloudShell` and `sts:GetCallerIdentity`** unconditionally — treat these as boilerplate every lab needs (CloudShell is how learners run the CLI track; `GetCallerIdentity` is how they confirm who they are / find their account id).
- Scope every other statement to **exactly this lab's resources** — S3: `Resource: ["arn:aws:s3:::sslab-*", "arn:aws:s3:::sslab-*/*"]` (prefix-scoped, not per-bucket-named, so it survives the account-id-suffixed bucket names). IAM: `Resource: "arn:aws:iam::*:user/lab/*"` — this is exactly why the `/lab/` path convention in §1 matters; the policy scoping and the resource-naming convention are the same mechanism.
- **A separate "attacker" identity (leaked keys) is a different scoping problem.** In `iam-privilege-escalation`, the exploit runs from the `pipeline-deployer` access keys minted by the CFN template itself (its own permissions, defined in the template's `DeployerPolicy` resource — intentionally over-broad, that IS the vulnerability). The learner's *console* `learnerPolicy` only needs to scope the **defender/admin seat** (remediate + verify + read the flag) — see the `_learnerPolicy` comment in that lab's lab.json: *"the attack runs from the SEPARATE leaked pipeline-deployer keys (their own perms), so this only scopes the admin/defender seat."* If your lab has this two-identity shape, don't try to fold the attacker's permissions into `learnerPolicy` — they belong in the template's IAM resources instead.

### Enforcement gates (in order of when they run)

1. **Build-time hard gate** — `app/scripts/build-lab-content.mjs`, the "Least-privilege gate" section: for every slug that has a `template.yaml` (i.e. is "ready"), it requires a non-empty `learnerPolicy` Statement array in the **engine-side** lab.json, and fails the entire build if missing. It also enforces a **size ceiling**: `MAX_BARE_POLICY_CHARS = 1000` on the bare (un-enveloped) JSON. The comment explains why 1000 and not the nominal 2048 STS limit: *"STS PACKS (compresses) session policies and the PACKED size — not the 2048-char plaintext limit — is the real ceiling... an ~1166-char bare doc already blew the packed budget."* Prefer resource-scoped wildcards (`sslab-*`) over long enumerated action lists to stay under this.
2. **Runtime soft warning** — `mintConsoleUrl` logs (does not block) if a lab has no policy.
3. **Manual authoritative verification** — `engine/verify-leastpriv.mjs`, run as `node verify-leastpriv.mjs <accountId> <slug>`. This is the ONLY check that actually assumes the role WITH the merged policy in a real account and probes allow/deny outcomes (it currently hardcodes probes for `sts:GetCallerIdentity` allow, `iam:ListUsers` allow, and denies for `dynamodb:ListTables`/`lambda:ListFunctions`/`iam:ListRoles` — **these probes are s3-lab-shaped**; for a new lab you should add probes specific to what that lab's policy allows/denies, not just run the script unmodified and assume a pass means anything).

## 4. instructions.md conventions (exact, from both live labs)

The file is markdown that gets pulled verbatim into `labInstructions[slug]` by the
build script, then re-parsed client-side by `app/components/lab-guide.tsx`. Every one
of the following is a hard parsing convention — get the syntax wrong and content
either doesn't render specially or (worse) silently merges into the wrong step/track.

1. **`<!-- ss:walkthrough -->` sentinel** — marks the boundary between the always-visible
   **Overview** (before) and the **Walkthrough** (after). This is a *security* boundary
   for paid labs: `app/labs/[slug]/page.tsx` ships only the overview to the client for
   `lab.ready && !lab.free` labs; the walkthrough is meant to be fetched later behind an
   entitlement check. **Free labs ship the whole file regardless.** If you omit the
   sentinel, the code falls back to splitting at the first `## Step` heading, and if
   there's no such heading either, the ENTIRE file is treated as overview (never gated)
   — so a paid lab without the sentinel or without `## Step` headings leaks its full
   walkthrough to anonymous visitors. Always include the sentinel explicitly.

2. **`## Step N — Title` headings** (em dash, exact pattern
   `/^##\s+Step\s+\d+\s*[—–-]\s*(.+?)\s*$/gm`, used identically in three places —
   `lab-guide.tsx`'s `STEP_HEADING_RE`, its `splitWalkthroughIntoSteps`, and
   `app/labs/[slug]/page.tsx`'s `extractStepTitles`, all three MUST stay in sync per
   the comment in the page file). Steps are numbered sequentially starting at 1. Any
   dash character (`—`, `–`, or `-`) works as the separator. The client renders one
   step at a time (step-through UI), each becoming its own segment with its own
   Console/CLI toggle.

3. **Track markers — `🖱️` (console) / `⌨️` (CLI)** — a line starting with the emoji
   opens a track block that runs until the next track marker, the next heading, or a
   `---` rule. Fenced code blocks (` ``` `) are respected — a `#` or emoji *inside* a
   code fence is never treated as a delimiter. The UI renders both tracks and lets the
   learner toggle between them (`TrackToggle` — "🖱️ Console" / "⌨️ CLI" tabs); content
   outside any track marker is "common" prose shown regardless of track.

4. **`:::refcard ... :::` fence** — a reference-card block (used once per lab so far,
   for "here are the real resource names the engine filled in" — see the `refcard`
   block in the S3 lab listing `DataBucketName`/`AssetsBucketName`/`auditor`). Rendered
   as a distinct visually-separated aside, not inline prose.

5. **`>>` breadcrumb lines** (console track only) — a line starting with `>>` becomes a
   clickable-looking nav-path breadcrumb, segments split on `›`, e.g.
   `>> S3 › data bucket › Permissions`. Leading indentation is preserved (stays inside
   its list item). This conversion (`chipify()`) runs on common + console text, **NOT**
   on CLI text (so a legitimate `>>` inside a bash heredoc or redirect is never
   mis-rendered).

6. **`[[Label]]` inline chips** (console track, same `chipify()` pass) — becomes a
   button-like "click this" chip, e.g. `Click [[Edit]], tick **all four** boxes, then
   [[Save]]`.

7. **Fenced code blocks** get a dark terminal wrapper with a copy button
   (`TerminalPre`/`CopyButton` in lab-guide.tsx) automatically — no special authoring
   needed beyond normal ` ``` ` fences, just don't rely on any specific language tag
   for behavior (only used for inline-vs-block detection: `language-` class + newline
   presence).

8. **Structure both existing labs follow** (not enforced by code, but consistent):
   `# Title` → level/free/duration/region line → "Before you start" orientation
   (S3 lab only — its intro predates the sentinel-split pattern, so check whether new
   labs should follow the *shorter* app/content version's structure, which starts
   straight at `## The scenario` since the orientation boilerplate is now handled by
   `LabIntro`/`EntitlementStatus` React components, not per-lab markdown) → `## Scenario`
   → `## Your mission` (bullet list mirroring `successCriteria`) → sentinel →
   `## Setup` (if there's an attacker identity to configure) → `## Step 1` … `## Step N`
   → `## Check your work` → `## Hints` → `## Cleanup` (always "nothing to do, aws-nuke
   wipes it").

   **Note the duplication:** `labs-platform/labs/<slug>/instructions.md` (engine copy)
   and `labs-platform/app/content/labs/<slug>/instructions.md` (app copy) are NOT
   identical in the existing labs — the app copy is the evolved/current one (uses
   `:::refcard:::`, `>>`, `[[ ]]`); the engine copy under `labs/` looks like an older
   plain-prose draft. Confirm with the codebase before assuming the engine copy of
   instructions.md is read by anything at runtime — from the file audit, only
   `template.yaml` and `lab.json` under `labs-platform/labs/<slug>/` are read by the
   engine (`labinfra.mjs`'s `deployStack`/`labMeta`/`labLearnerPolicy` all join
   `__dirname, "labs", labSlug, ...`); `instructions.md` under `labs-platform/labs/`
   appears to be author-reference/dead weight rather than a live artifact. Treat the
   **app-side** `instructions.md` as the one that actually ships to learners.

## 5. H11 — catalog duplication (documented, NOT fixed here)

The lab list/metadata exists in (at least) **three independent, hand-synchronized
locations**, plus two cosmetic special-cases:

1. **`labs-platform/app/content/labs/<slug>/lab.json`** — the nominal "source of truth"
   per its own comment in `app/lib/labs.ts` (*"SINGLE SOURCE OF TRUTH = each lab's
   app/content/labs/<slug>/lab.json"*). Feeds the generated `lab-catalog.ts` /
   `lab-content.ts` via `build-lab-content.mjs`. Has `title`, `level`, `free`, `tags`,
   `estimatedActiveMinutes`, `summary`, `successCriteria`, `hints`.
2. **`labs-platform/labs/<slug>/lab.json`** — the **engine's** copy, read directly by
   `labinfra.mjs` at runtime (not generated, not derived from #1 — a fully separate
   hand-authored file). Overlaps on `slug`/`title`/`level`/`free`/`tags`/
   `estimatedActiveMinutes`/`summary`/`successCriteria`/`hints` (near-identical to #1
   in both existing labs, but nothing enforces they match), and additionally carries
   `learnerPolicy` (which #1 does NOT have — the engine copy is the only one the
   least-privilege gate checks).
3. **`shieldsync-website/lib/site.ts` → `AWS_LABS`** — the marketing site's own array,
   explicitly commented as mirroring #1 (*"CANONICAL shared fields ... mirror each
   lab's labs-platform/app/content/labs/<slug>/lab.json — keep them in sync"*) but
   physically copy-pasted, plus marketing-only fields (`desc`, `added`, `skills`,
   `roles`) that don't exist anywhere else.

Two smaller, lower-stakes duplications worth knowing about but not counted in the "3":
- `shieldsync-website/components/labs-wizard.tsx`: `FREE_SLUG` constant, hardcoded slug string.
- `shieldsync-website/app/aws-security-certification/page.tsx` (~line 264): hardcoded `slug === "s3-misconfiguration-audit"` conditional for a "FREE" badge.
- `labs-platform/engine/handler.mjs` and `server.mjs`: `PRIMARY_LAB` constant (warm-pool pre-provisioning target only — cosmetic to correctness, real to first-launch latency).

**What actually breaks if these drift:** #1 vs #2 drifting on `level`/`free` doesn't
break anything visibly — the engine's access-rules (`rulesFor`) reads `level`/`free`
from #2 only, so #1's copy of those fields is purely informational/legacy at this
point given #1 no longer needs them for anything except being the generator input for
the UI catalog. #1 vs #3 drifting is the one a learner would actually notice — e.g. the
marketing site's duration/tags/title going stale against what the lab page itself
shows.

### Proposed consolidation (NOT implemented — proposal only)

Collapse to **one authored file per lab**: `labs-platform/labs/<slug>/lab.json`
(the engine copy, since it's the one with the strictest existing gate — the
least-privilege check already treats it as authoritative for `learnerPolicy`). Make it
carry every field currently split across #1/#2 (add `successCriteria`/`hints`/
`summary`/marketing fields directly there, or keep marketing-only fields (`desc`,
`added`, `skills`, `roles`) in a small separate marketing-only overlay file since
they genuinely don't belong in the engine's runtime-read file). Then:
- Point `build-lab-content.mjs`'s `CONTENT` constant at `labs-platform/labs/` instead
  of `app/content/labs/`, so the app generates its catalog from the same file the
  engine reads — deletes duplication #1 vs #2 entirely.
- Either (a) have `shieldsync-website` import the generated `lab-catalog.ts` from
  `labs-platform/app/lib/` directly (if the two are in a shared workspace / can share
  a package), or (b) write a small script that generates `AWS_LABS`'s canonical fields
  from the same source and fails the marketing build if a lab.json exists with no
  matching `AWS_LABS` entry (mirroring how `build-lab-content.mjs` already fails the
  build for a ready lab with no `learnerPolicy`) — cheaper than a full monorepo import
  if the two repos need to stay independently deployable.
- This is a real refactor (touches the build script, both repos, and needs a
  migration pass over the two existing labs) — scope it as its own ticket, don't fold
  it into adding lab #3.

## 6. Step-by-step: adding lab N

Assume slug `my-new-lab` throughout.

### 0. Copy the template
```
cp -r labs-platform/labs/_template labs-platform/labs/my-new-lab
```
You'll also need to hand-create the app-side pair (the template only ships the
engine-side skeleton plus a stub for the app side — see the template's README).

### 1. Author the scenario (`template.yaml`)
- Design a deliberately-misconfigured mini-environment with 2-4 fixable flaws (both
  existing labs sit in that range — enough to teach something real, not a scavenger hunt).
- Follow every convention in §1: `LabSlug` parameter matching the directory name,
  `ShieldSyncLab` tag on every taggable resource, `/lab/` path for learner-facing IAM
  identities, `sslab-` prefix for other named resources, `DeletionPolicy: Delete` +
  `UpdateReplacePolicy: Delete` everywhere, a seeder Lambda that **empties** any bucket
  it seeds on stack Delete.
- Keep cost near zero — no idle compute, no NAT gateways, no RDS.
- Add an `Outputs:` block with the real resource names + a console deep-link.
- **Local validation before wiring anything else in:**
  `aws cloudformation validate-template --template-body file://template.yaml`,
  then a real deploy test (`engine/test-deploy-labs.mjs` exists — check whether it's
  generic across labs or S3-specific before assuming it "just works" for a new slug).

### 2. Author both `lab.json` copies
- Engine copy (`labs-platform/labs/my-new-lab/lab.json`): `slug`, `title`, `level`,
  `free`, `track`, `tags`, `estimatedActiveMinutes`, `accessWindowHours`,
  `idleAutoStopMinutes`, `summary`, `learningObjectives`, `successCriteria` (with
  `id`/`description`/`verify` — `verify` is prose documentation of the check, not
  executable), `hints`, `_learnerPolicy` (comment) + `learnerPolicy` (Statement[] —
  see §3), `estimatedAwsCostInr`, `deploysToRegion`.
- App copy (`labs-platform/app/content/labs/my-new-lab/lab.json`): same shape, minus
  `learnerPolicy`/`_learnerPolicy` (not read from here) — keep every overlapping field
  **identical** to the engine copy until/unless the H11 consolidation happens.
- **`successCriteria[].id` values are load-bearing** — they must exactly match the
  `id`s your `graders.mjs` function will return in step 4.

### 3. Wire the learnerPolicy
- Write the `Statement[]` per §3: always include `CloudShell` + `sts:GetCallerIdentity`
  boilerplate, then scope every other Allow to exactly the resources your template
  creates (prefix-scoped where the resource name has an account-id suffix).
- Check the bare-JSON size stays comfortably under 1000 chars (the build gate's
  ceiling) — run `node -e "console.log(JSON.stringify(require('./labs-platform/labs/my-new-lab/lab.json').learnerPolicy).length)"` as a quick gut-check before a full build.

### 4. Write the grader
- Add `async function gradeMyNewLab(creds, accountId) { ... }` to `graders.mjs`,
  following the pattern in §2: probe live state, distinguish absence-errors (fixed)
  from real errors (`unknown`), return `[{id, description, passed, ...unk(err)}]` with
  ids matching `successCriteria`.
- Add the dispatch line: `else if (labSlug === "my-new-lab") criteria = await gradeMyNewLab(creds, accountId);`
- If your lab touches a service other than S3/IAM, import that service's SDK client at
  the top of `graders.mjs`.

### 5. Write instructions.md (app-side; the one that ships)
- Follow every convention in §4: sentinel, `## Step N — Title` headings, `🖱️`/`⌨️`
  track markers, `:::refcard:::` for the "here are your real resource names" block,
  `>>`/`[[ ]]` chips in console-track steps.
- End with `## Check your work`, `## Hints`, `## Cleanup` (standard "nothing to do,
  aws-nuke wipes it" line unless your lab genuinely has learner cleanup steps).

### 6. Regenerate + register
```
cd labs-platform/app
node scripts/build-lab-content.mjs
```
This regenerates `lib/lab-catalog.ts` + `lib/lab-content.ts` AND runs the
least-privilege gate (will hard-fail the command if `learnerPolicy` is missing/oversized).
Then hand-add the `AWS_LABS` entry in `shieldsync-website/lib/site.ts` (§1, registration
point #3).

### 7. Test — verification gates (all four must pass before calling it done)

1. **Deploys clean.** Deploy the stack into a real (or the sandbox) account; confirm
   `CREATE_COMPLETE`, all `Outputs` populated, no orphaned resources on a subsequent
   `DELETE`. Re-deploy from scratch a second time (simulates the pool recycling the
   account) to catch any resource-naming collision.
2. **Grades correctly on the BROKEN (freshly-deployed) state.** Immediately after
   deploy, call the grader — every criterion should read `passed: false` (nothing
   fixed yet). A criterion that passes on the broken state means your grader logic is
   inverted or checking the wrong thing.
3. **Grades correctly on the FIXED state.** Manually perform every remediation step
   from your own instructions.md, then re-grade — every criterion should flip to
   `passed: true`. Do this via BOTH the console track and the CLI track at least once
   each, since they should converge on the same account state.
4. **Session policy blocks out-of-scope actions.** Run
   `node engine/verify-leastpriv.mjs <accountId> <slug>` — but first edit the probe
   list in that script to match YOUR lab's actual allow/deny boundary (the shipped
   probes are S3-lab-shaped, per §3). Confirm the learner's minted console session can
   do everything the walkthrough asks and nothing more (spot-check a couple of
   plausible temptations — e.g. can they enumerate unrelated services, can they touch
   the `ShieldSyncLab*`/`OrganizationAccountAccessRole` roles — both should be denied
   by the fixed guardrail regardless of what you wrote).

Only after all four pass should the lab be considered launch-ready (`ready: true` is
actually computed automatically — it's `existsSync(template.yaml)` — so technically the
UI will show it as ready the moment you commit the template; treat these four gates as
your own pre-commit bar, not something the platform enforces for you).

## Learner-facing copy rules (owner directive, 2026-07-03)

Before a lab flips to ready:true, sweep ALL learner-visible text (instructions.md,
UI strings, catalog blurbs) for internals. Learners only see what matters to THEM.

Never name in learner-visible copy:
- wipe tooling (aws-nuke) -> say "your account is wiped clean automatically"
- account pooling / "the pool" -> say "free seats are limited" / "at busy times"
- engine / Session Engine / Lambda / DynamoDB / Cloudflare / wrangler internals
- Cognito -> say "sign in with Google"
- internal AWS account IDs or role names that are not part of the lesson

AWS concepts that ARE the lesson (IAM, S3, STS, CloudFormation-as-topic) are fine.
Code comments are exempt (not learner-visible). Rationale: learner clarity first,
and no free recon for attackers about our teardown/infra stack.
